# /team — конвейер команды агентов

Ты — оркестратор. Агенты не общаются напрямую: ты передаёшь выводы одного
на вход другому (полные, не пересказ по памяти — цитируй план и отчёты).
Каждый агент стартует холодным: в промпт всегда включай текст задачи и
напоминание прочитать CLAUDE.md и docs/design-systems.md.

## Конвейер

### 0. Приём задачи
Сформулируй задачу в 2–3 предложениях и критерии готовности. Если в задаче
есть развилка, которую нельзя решить самому (масштаб, платный сервис,
изменение публичного контента с фактами) — один AskUserQuestion, не больше.
Создай задачи через TaskCreate: по одной на этап конвейера + по одной на
крупный блок реализации. Далее веди статусы TaskUpdate по ходу.

### 1. План — агент architect
Запусти `architect` с полным текстом задачи. Получи план.
Оцени сам: полный ли, реалистичный ли, нет ли конфликта с правилами.
Пробелы закрой через SendMessage архитектору (не додумывай молча).
План утверждён — зафиксируй его текст в задаче (TaskUpdate description).

### 2. Реализация — агент(ы) implementer
Один блок работы — один implementer; в промпт: задача + относящаяся часть
плана целиком + критерии готовности.
Параллельно только если блоки не пересекаются по файлам — тогда каждому
`isolation: "worktree"`, а после завершения смёржи их ветки сам.
Последовательные блоки — без worktree, в основном дереве.

### 3. Ревью — агент reviewer, затем /code-review
Запусти `reviewer` с дифом (git diff --stat + git diff по ключевым файлам)
и отчётом implementer (включая пути скриншотов).
Блокеры и «надо исправить» → верни соответствующему implementer через
SendMessage (или почини сам, если правка тривиальна). Цикл до чистого
ревью. Затем прогони скилл `/code-review` по диффу как независимую
проверку на баги; найденное — почини.

### 4. Верификация
`pnpm build` зелёный; скриншоты изменённых экранов (desktop 1440 +
mobile 390) сняты и просмотрены тобой лично (Read), не только агентами.

### 5. Документация — агент doc-writer
Запусти `doc-writer`: текст задачи, финальный план, git diff --stat,
итоги ревью. Проверь его диф — документация не должна противоречить коду.

### 6. Сдача
- Коммит(ы) в рабочую ветку по конвенции репозитория, push.
- Merge ff в `main` + push — ТОЛЬКО если пользователь просил выкатить на
  лайв (или явно просил в этой сессии ранее «сразу на лайв»); иначе спроси.
- После пуша в main: проверь статус GitHub Actions ран(а); упавший — чини.

### 7. Итоговый отчёт пользователю
Одно сообщение: что сделано и где смотреть (ссылки на страницы),
ключевые решения, что говорит документация (путь к docs/features/<slug>.md),
скриншоты через SendUserFile, открытые вопросы. Русский язык, без жаргона
конвейера — пользователь видит результат, а не процесс.

## Правила оркестратора

- Не запускай следующий этап, пока предыдущий не принят тобой явно.
- Пустой отчёт агента или «всё сделал» без диффа — не результат: запроси
  детали через SendMessage.
- Ты отвечаешь за итог лично: агенты могут ошибаться, финальная проверка
  сборки и скриншотов — всегда твоя.
